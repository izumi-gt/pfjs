## プロジェクト全体像（不変）
社会福祉法人の計算書類（全12様式）をPDFから自動パースし、SQL移行前提の統合DB
「決算情報データベース」へ取り込む。最終目標 約5万法人。現在 手元440社で品質改善中。
3層分離: パーサ（形のゆれ）／レジストリ（名前のゆれ＋採番）／エンジン（意味の検算）。

## 現在の状態（ここまでの到達点）
- 440社を取り込み済み（`out_csv/<法人番号>/` にCSV、`_scan_summary.csv` に全社サマリ）。
- 2社（神奈川・あしたか）製だったマスタを、別表正典ベースでCF/PL大区分46件拡充済み。
- タイプ4の未知大区分を拠点別擬似親 `@LOC:{loc}` で採番する修正を adapter_kyoten4.py に、
  空/擬似親parentを許容する修正を build.py の post_checks に適用済み。
- 結果: FK・PK有 371社 → 98社、FK/PK=0 が 342/440社まで改善。
- **残る98社の主キー重複**が本タスクの対象。内訳は設計指示書§2参照。

## あなたのゴール
設計指示書に従い、fact主キー重複を**主キー設計の見直しで根本解決**する。
1. 重複5パターン（集計行配下/通常parent/幹コード直emit/擬似親@LOC/parent空）の
   実態を、診断スクリプトとパーサ抽出ダンプで確認する。
2. 設計案を**最低3案**（指示書の案A/B/C等）、トレードオフ付きで提示する。
3. 採用案の合意を得てから実装する（**いきなりコードを書かない**）。
4. 神奈川基準線（fact 3,326 / sha256 / NG=0）を保ったまま実装し、440社で検証。

## 神奈川基準線の検証コード（エンジンを触ったら毎回実行）
```python
import hashlib
from shafuku_db_engine.corps.kanagawa import CORP
from shafuku_db_engine.ingest import ingest
ing = ingest(CORP)
blob = '\n'.join('|'.join(str(x) for x in row) for row in ing.fact)
sha = hashlib.sha256(blob.encode('utf-8')).hexdigest()
assert len(ing.fact) == 3326
assert sha == '0aa61d8b7c250c447e39688d9b8b38fe665c6f3812fbdc5bc490f5101f102e54'
print('OK')
```
これが通らなくなったら、その変更は採番順/出力順を変えている（sha破壊）。

## 重要な制約（厳守）
- **会計値を消さない・潰さない。** 同一科目の複数出現は会計的に正当（BSの資産/負債で
  同名、計算書の小計繰り返し等）。「区別して全保持」が正解。重複を削るのは誤り。
- **神奈川基準線を壊さない。** 採番順・出力順を変えず、区別次元は付加する形で。
- **まず設計案を比較提示してから実装。** 主キー設計はプロジェクトの根幹かつ
  「案2 identity統一フェーズ」に関わる。場当たり実装を避け、合意を取る。
- 5万社展開での拡張性を意識する。

## 最初にやること
1. `主キー重複_設計指示書.md` と本プロンプトを読む。
2. zip展開→神奈川基準線が現状で通ることを上記コードで確認。
3. `pk_dup_breakdown.csv` と診断スクリプトの結果で重複の全体像を把握。
4. 幹コード直emit(699)・集計行配下(1,415)の実態を、該当社のパーサ抽出ダンプで確認
   （誤抽出なのか正当な複数出現なのかを切り分ける。誤抽出ならパーサ修正、
   正当ならキー設計で区別）。
5. 設計案を提示し、合意後に実装・検証。

完成したら、採用設計・実装差分・440社での重複減少結果を依頼元チャットに報告してください。
