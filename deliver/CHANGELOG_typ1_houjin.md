# CHANGELOG — タイプ1（法人単位 1-1 / 2-1 / 3-1）アダプタ実装【分岐A】

本体チャットの最新版（`shafuku_kyoten4_adapter.zip`）を土台に、タイプ1のアダプタを実装した。
あしたか太陽の丘の 1-1/2-1/3-1 を **validate NG=0** で取り込み、神奈川基準線
（fact 3,326行・sha256 `0aa61d8b…f102e54`・NG=0）は**完全非破壊**で維持。

---

## 新規追加ファイル（衝突なし）

### shafuku_parser/adapter_houjin.py
タイプ1の橋渡しアダプタ本体。
- `build_corpdata(pdf_paths, corp_no, corp_name, fiscal_year, masters, ...)` →
  `(CorpData, registries, review_rows, warnings)`。
- 1-1/2-1（階層なし・全行 global stem）: `resolve_global` のみで `{code:(m0,m1,m2)}` を構築。
  local 採番は不要（タイプ1は明細を持たないため review はゼロ）。
- 3-1（BS）: **中区分セクション文脈つき解決** `_BSSectionResolver` を実装。
  - 背景: BS_MASTER は `土地/建物/定期預金/貸倒引当金/その他の固定資産` が
    基本財産・その他の固定資産・流動資産にまたがって**同名で複数回**出る。
    registry の `global_by_norm` は名前単独キーなので同名は最後の1つに潰れる
    （タイプ4の拠点別BS=大区分のみではこの問題に当たらなかった）。
  - 対応: 共有の registry/masters は触らず、アダプタ内で BS_MASTER から
    `(中区分, 正規化名) -> code` 表を作り、行を文書順に走査して現在の中区分を追跡。
    `その他の固定資産` のヘッダ(indent小)→明細(indent大)の二重出現も
    「セクション遷移時にヘッダ枠を消費」して弁別。
- io導出・本部拠点剥がしは adapter_kyoten4 と同方針（タイプ1は本部拠点なし）。
- ノイズ行（様式タイトル・縦書きラベル断片・純資産下の `0`）は global 未解決として
  スキップ。**値を持つのに未解決の行は warnings に記録**（無人完走は止めない）。

### shafuku_parser/run_houjin.py
タイプ1 無人完走ランナー（検算→隔離→CSV出力）。
- run_kyoten4 の CSVヘルパ（`_write_csv` / `_global_account_rows` / `_dedup_concepts`）を
  import して流用。タイプ1は拠点・事業区分が無いので、NGは様式(1-1/2-1/3-1)単位で隔離。
- 出力 CSV（UTF-8 BOM）: fact_financial / dim_* / review_queue / quarantine / parser_warnings。

## 共有ファイルへの変更（1ファイルのみ・本体マージ対象）

### shafuku_parser/extract_houjin.py — 縦書きマージン単字の除去（追加のみ）
**変更内容**: `_strip_vertical_margin_words(words)` を新設し、`extract_houjin` 冒頭の
`group_rows` 直前に1行 `words = _strip_vertical_margin_words(words)` を挿入（純粋に追加。
既存ロジックの書き換えなし）。

**理由（バグ）**: PL/CF の 1-1/2-1 で、左端の縦書き区分/収支ラベル列（`増` `の` `事` 等の単字）が
独自の top を持ち、`group_rows(y_tol=3.0)` で**値数字の行を奪っていた**。
具体例: `事務費`(top 298.0) の値 `231,370,166`(top 297.0)/`-21,248,977` が、直上の
マージン単字 `増`(top 294.9) の行に吸われ、`事務費` には前年度 `252,619,143` だけが残る
→ `PL 2-1 SO差額`（サービス活動外差額）NGの原因。同様に `サービス活動外費用計` も欠損。

**修正の作法（過学習回避・座標のみ）**:
- 「2文字以上のラベル語」の x0 最頻値 body_x0 を実測し、`x0 < body_x0 - 6` の帯に居る
  **単一文字** word のみをマージンとして除去。
- 左寄せの計・差額行（`経常増減差額` 等、x0が浅い）は2文字以上なので誤除去しない。
  実科目名で1文字のものは存在しない（最短 `土地`/`権利`=2字）ため単字判定は安全。

**基準線への影響**: 無し。`extract_houjin` は神奈川の手書き corps では使われないため、
fact 3,326行・sha256・NG=0 は不変（確認済み）。engine/naming/registry は無変更。

## 実証結果（あしたか太陽の丘 8080105000129）

| 様式 | コード数 | fact | NG | PDF実額突合 |
|---|---|---|---|---|
| 1-1 (CF) | 53 | — | — | 事業活動収入計/当期末残高 一致 |
| 2-1 (PL) | 53 | — | — | サービス活動収益計/次期繰越 一致 |

- review_queue: 0件（タイプ1は明細なし）。warnings: 0件（マージン修正後）。
- BSバランス: 負債合計 328,829,693 + 純資産 3,772,216,981 = 資産合計 4,101,046,674（一致）。
- **タイプ1＋タイプ4 結合検算**: 同一法人の 1-1/2-1/3-1 + 1-4/2-4/3-4 を1つの CorpData に
  載せて `validate` → **NG=0**、fact 9,425。様式間突合・計算書またぎ(BS↔PL)も green。
  → タイプ1の抽出がタイプ4（検証済み）と整合していることの強い裏付け。

### 補足: タイプ1とタイプ4「合計」の差は正常（バグではない）
タイプ1法人合計とタイプ4全拠点総和を直接比較すると、収入計・資産合計などが一致しない。
これは**内部取引消去（拠点区分間繰入金・貸付/借入金）**の差で、タイプ4は gross 合算、
タイプ1は net（消去後）であるため。当期収支差額合計・当期活動増減差額・純資産合計など
消去が相殺される最終値は完全一致する（確認済み）。正しい突合相手はタイプ2の法人合計列。

## 本体マージ時の申し送り

- **共有ファイル変更は `extract_houjin.py` の1点のみ**（追加のみ）。他分岐（B=segwari, C=segwari3）が
  同ファイルを触らなければ衝突しない。触る場合は本 `_strip_vertical_margin_words` と統合検討。
- engine（ingest/validate）・naming・registry は**無変更**＝基準線sha256一致を保証。
- 新規2ファイルは別名のため衝突なし。
- BSの同名弁別ロジック（`_BSSectionResolver`）は adapter_houjin 内に閉じている。
  タイプ3(3-3)も BS小区分を持つため、分岐Cが同種の同名問題に当たる可能性あり。
  その場合は本リゾルバの考え方（中区分×文書順）を流用できる旨を共有する。
